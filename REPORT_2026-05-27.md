# KM27 — State of the Game Report

**Date:** 2026-05-27 · **Audit type:** static-analysis review with targeted bug-fix pass · **Author:** Claude (session)

---

## 1. Where the project is, at a glance

KM27 is in a strong place. Phases 0–7 are done and the full week loop — Title → Knight chooser → Roster → Planning → Tick → Pre-Battle Review → Battle Log → Weekly Summary → next week — is end-to-end playable, with Tournaments, Grand Tournament, and Game Over / Run Win endings all wired. What followed the Phase-7 finish line was an extended polish-and-systems-expansion sprint that has, by the count in ROADMAP, shipped roughly two months of substantive content and structural work without ever entering formal Phase 8 balance.

The codebase is 18,054 lines of GDScript across 69 files. It is well-organised: data lives in `scripts/data/`, stateless rules in `scripts/systems/`, autoload singletons in `scripts/autoload/`, screen controllers in `scripts/screens/`, and shared widgets in `scripts/ui/`. The split is observed in practice — `Combat`, `Resolution`, and `BattleEvent` genuinely do not read `GameState` directly, which is the single architectural rule that pays the most dividends across this kind of project.

Recent layers worth calling out as load-bearing:

- **Staged stat development** (`Stats.add_progress` + the `progress` / `momentum` / `developing` dicts on each `Stats` instance). This is the right architecture for FM-style growth — fractional accumulator, headroom factor, two recency dicts for the bright vs. small arrow. It quietly fixes a much wider design problem ("training feels lottery-shaped") in a way that won't need to be revisited.
- **The data-driven event engine** (`StoryEventDB` ~92 entries, `AwayModeDB` ~10, `CombatEventDB` ~5). Adding content is now mostly pure data, which means content velocity stays high through Phase 8 and beyond without code review.
- **The FM week processor overlay**. Conceptually small but disproportionately important to the *felt* texture of the game — a week now has weight.
- **Save manager** robust and back-compatible. Old saves load; new fields default empty.

The single biggest unaddressed item is, as ROADMAP has noted across nearly every recent entry, **Phase 8 balance**. Almost everything that exists today is sitting on placeholder numbers.

## 2. What this session did

I read the project carefully end-to-end — ROADMAP, CLAUDE.md, the autoloads, the resolution / combat / sim / tick layer, the planning + pre-battle screens, the save manager, and a sampling of the data files. I could not run Godot headless from my sandbox (the binary lives on your Windows host), but you ran it during the session and reported `[KM27] Title ready.` cleanly, which is the all-clear signal for parse / autoload wiring.

The audit found four real bugs and one QoL gap. All five were fixed in a single batch on this branch.

### Fixes shipped

1. **`current_event_uses_formation()` extended.** The function only returned true for `bandit_ambush` inside `BATTLE_EVENT`, even though `village_raid`, `tavern_riot`, and every `CombatEventDB` combat-template entry also resolve through `_player_cus_home` and a formation editor. The Pre-Battle Review's default-formation seeder uses this predicate, so on those weeks the Tactics-tab default never auto-applied and the player had to slot every unit manually. Fixed in `scripts/autoload/game_state.gd`.

2. **Forecast / Resolution mismatch on the 0.75× modifier.** `pre_battle_review._is_home_battle()` only honoured the non-Defend penalty for `bandit_ambush`. The same set of sub-types as fix 1 actually applies the penalty at resolve time, so the forecast was over-optimistic on `village_raid`, `tavern_riot`, and `CombatEventDB` events. The forecast now matches what the sim will actually do.

3. **`maintenance_debt` sticky flag.** `Tick._apply_gold_maintenance` set `gs.maintenance_debt = true` on a short week but never cleared it. `Chronicle._event_line` reads the flag every week — so once a player went short once, the chronicle line *"The ledger fell short this week. The household will feel it."* appeared on every subsequent week's chronicle, even when wages were paid in full. The flag is now per-week truth.

4. **`CombatSim` sort comparator non-determinism.** The initiative-jitter lambda was calling `RNG.randi_range(0, 2)` *inside* the sort comparator. The same pair `(a, b)` could compare both ways across consecutive calls, which violates sort invariants. The jitter is now rolled once per combatant before the sort and stored on a new `CombatUnit._init_jitter` field; the comparator is a pure stable key compare. Same combat feel, deterministic ordering.

5. **Planning state survives scene changes.** `_pending_tasks` was a local Dict on `planning.gd`. Clicking a knight's name to open Knight Overview reloaded Planning on return, the local dict re-initialised empty, and `_default_pending_tasks()` quietly defaulted everyone back to Defend — wiping any training plan the player had set up. State moved to `GameState.pending_tasks`, cleared by `_clear_week_buffers()` so each new week still starts fresh.

ROADMAP's Progress Log has a newest-first dated entry covering this batch.

## 3. The codebase, where it's strong

**Architectural discipline.** The autoload / data / system / screen split is real, not nominal. `Resolution` is the only system that mutates state during combat resolution. `Combat`, `BattleEvent`, and `OutcomeBracket` take inputs and return Dictionaries — pure, testable, and trivial to retune. `Tick` is similarly hands-off — it reads `gs.roster`, mutates units in place, returns a result Dictionary the UI renders. Most file-level touch points are documented in CLAUDE.md's cheat-sheet, which is a living thing rather than an aspirational README.

**Determinism.** All gameplay randomness routes through the `RNG` autoload. `WorldGenerator.generate(seed)` stamps the seed onto the World, `SaveManager` round-trips it, and the same seed reliably reproduces the world plus the starting roster. That's the kind of invariant that becomes hard to add later but trivial to maintain if you start with it — and KM27 has.

**Data-driven content.** `StoryEventDB`, `AwayModeDB`, `CombatEventDB`, `TraitPool`, and `HousePool` all share a clean shape: a `EVENTS` / `MODES` / `HOUSES` dict with weighted entries and effect primitives. Adding a new chronicle event, away mission variant, or combat sub-type is roughly "add a dict entry." That's the right structure for the next year of content additions.

**Save format is back-compatible.** Every recent feature (item stockpile, oath_kind, house_leans, trait_id, staged-dev progress / momentum / developing) appears in `save_manager.gd` with a default for old saves. This is the kind of discipline that lets players keep their run when they update.

**Show-not-tell where it matters.** PA stays hidden everywhere it should — dev toolbar shows it, no player-facing UI does. Stat caps are obfuscated through the band descriptor system. House leans are deliberately implicit. The new dev arrows do *exactly* the right thing: they fade as a stat approaches PA without ever exposing the number.

**The FM week processor is excellent.** It transforms a one-click commit into a paced, readable beat by beat narration of the week. Pause-on-notable beats is the right design.

## 4. Where the codebase is brittle, or where work is half-finished

### 4.1 Dead / nominal income sources

`gold_income_sources` includes `tournament_prize` and `expedition_trade` keys that are never written anywhere outside `start_run()` and the save manager. The cashflow display hides zero-valued sources, so they're invisible — but they're dead scaffolding. Either wire them up (a tournament win that adds, say, `+5` per week for the next 4 weeks would be a nice ongoing reward) or remove them.

### 4.2 ResourceBundle / ResourceDB dual system

This is documented in CLAUDE.md as deliberate, and I won't second-guess the call. It is worth noting that the longer this lives, the more new code accidentally reaches for the legacy triple (`wood / fibres / copper_ore`) instead of the new inventory keys (`logs / plant_fibres / copper_ore`). Worth scheduling a dedicated migration session before it grows another six callers.

### 4.3 Raw material sourcing

CLAUDE.md and ROADMAP both flag this. The crafting recipes work; the *inputs* don't really flow from gameplay yet because most raw materials still come from pillage/assault loot rather than gather/mob drops. This is the single biggest gap between "the systems exist" and "the systems compose into gameplay."

### 4.4 Oath break

The honour side is wired (`OathLedger` grants a hidden +3 PA on aligned action). The break side — penalties on violation — is not. With honour wired, break is the natural follow-up and would give oaths real mechanical bite.

### 4.5 Epithet trigger coverage

Most points fire; CLAUDE.md acknowledges "not every event yet." A pure-data sweep across `StoryEventDB` outcomes to add `grant_epithet` calls where they read right.

### 4.6 Stat development pace is unbalanced

`DEV_PACE = 4.5` (nominal weeks per point at full rate) and the headroom factor between `DEV_MIN_FACTOR = 0.12` and `1.0` are placeholders. The whole system was designed to be tunable from a few constants — but it hasn't been played long enough to know if 4.5 weeks/point is too slow (player feels stalled) or too fast (the dev arrows are always lit, so they stop reading as a signal). This is *the* Phase 8 number.

### 4.7 `_pending_tasks` and the `pending_*` cluster

Now that I've moved `_pending_tasks` onto `GameState`, the planning week's transient state lives partly there and partly in local screen state (`_expedition_party` is still local). It would be cleaner to centralise *all* per-week-pre-commit state on GameState — but right now the only one that actually matters across scene changes is `pending_tasks`, so I held the scope.

## 5. Suggested next steps, in rough priority order

These map to ROADMAP's "Likely Future Directions" but with my honest weighting after the audit.

### Priority 1 — Phase 8 balance pass (start it)

Genuinely the most valuable next step. Three concrete sub-steps:

1. **Run two full playthroughs end-to-end.** A "play to lose" run (don't optimise) and an "optimised" run. Capture week reached, outcome, and the felt issues — what stalls, what runaway-wins, what feels coin-flippy.
2. **Ground-truth `DEV_PACE`.** From the FM-style staged-dev system, the single biggest knob. Tune until a typical knight visibly improves across a full year of training without feeling like they crystallise too fast.
3. **Ground-truth the enemy power multipliers** (`× 3` on pillage, `× 4` on home, the castle curve, the new `CombatEventDB` `home_battle` template enemies). The sanity check in GDD §13 still holds; verify it does in practice.

This is where every other suggestion downstream gets context.

### Priority 2 — Wire raw-material sources

The crafting tab works but its inputs don't flow yet. Two surgical additions:

- **Mob drops on combat-event wins.** `CombatEventDB` already has `reward_kind` — add a `mob_drop_table` entry per template so a goblin warband can drop plant fibres, harpy raids can drop feathers (a new T1), etc.
- **Specific tile types yielding correctly on gather.** Mountains should yield raw ore on gather (the GDD §4 adjacency footnote that's deferred). Forest tiles should yield logs reliably, not the legacy `wood`.

Both unlock the Crafting tab as a meaningful loop rather than a tier-tree showcase.

### Priority 3 — Oath break

The honour side is wired; the break side is one symmetric system away. A unit whose behaviour violates their oath this week gets a hidden -2 or -3 PA penalty and a small chronicle note ("X strayed from his oath this week"). One file (`OathLedger`), small surface area, completes a mechanic the player can already sense exists.

### Priority 4 — Epithet trigger coverage sweep

Mostly data; mostly low-risk. Read through `Resolution` for any beat where a unit did the dramatic thing and check `grant_epithet` got called. Pair with any `StoryEventDB` outcome that should award one.

### Priority 5 — More content (data-driven catalogs)

Once Phase 8 has shown what's working, the data layers will absorb effectively unlimited content with no code review. `StoryEventDB` could comfortably grow to 200+ entries. `AwayModeDB` could hit 20. Add weapons, add armours, add traits. None of this is urgent but all of it is cheap.

### Priority 6 — Cleanup (low priority but the right time)

- Decide on `tournament_prize` / `expedition_trade` — wire or delete.
- Schedule the ResourceBundle → ResourceDB migration as a focused session.
- Consider centralising the other per-week-pre-commit state (`_expedition_party`, the formation picks) on `GameState` for consistency with the new `pending_tasks` location.

### Out of scope (per GDD §17, intentionally)

Squire promotion / recruitment / morale, more formations beyond 4-0-0, more enemy variety beyond tier-1. These are documented post-MVP and shouldn't pull focus until Phase 8 ships.
